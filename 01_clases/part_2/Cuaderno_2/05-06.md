## Tema: Patrones Estratégicos DDD y Arquitectura CLEAN vs Hexagonal

### 1. Patrones Estratégicos de DDD
A diferencia de los patrones tácticos (que se aplican directamente en el código para construir el modelo), los patrones estratégicos se centran en cómo organizar el sistema a gran escala y cómo manejar la complejidad del negocio.

* [cite_start]**Enfoque Estratégico y Subdominios:** Dado que no todo el software es igual de importante, DDD propone dividir el sistema en Subdominios para identificar el Dominio Principal (*Core Domain*)[cite: 1]. [cite_start]Este núcleo es la parte que otorga la ventaja competitiva a la empresa[cite: 1].
* [cite_start]**Lenguaje Ubicuo (Ubiquitous Language):** Es el idioma común compartido por todo el equipo (desarrolladores y expertos del negocio) que utiliza términos específicos del dominio[cite: 1]. [cite_start]Su objetivo principal es evitar traducciones erróneas entre lo que el negocio necesita y lo que el desarrollador programa[cite: 1].
* [cite_start]**Contextos Delimitados (Bounded Contexts):** DDD define fronteras explícitas donde un modelo y su Lenguaje Ubicuo son completamente válidos[cite: 1]. [cite_start]Esto permite que una misma palabra (por ejemplo, "Artículo") tenga significados, atributos y comportamientos distintos en diferentes partes del sistema (como Inventario vs. Facturación) sin causar conflictos[cite: 1].

### 2. Repositorios y Persistencia (ORM)
* [cite_start]**Patrón Repositorio:** Actúan como una ilusión de una colección en memoria de todos los objetos de un tipo específico (estrictamente enfocados en *Aggregate Roots*)[cite: 1].
* [cite_start]**Propósito:** Median directamente entre el modelo de dominio y la base de datos[cite: 1]. [cite_start]Su trabajo es guardar y recuperar agregados completos[cite: 1].
* [cite_start]**Aislamiento Técnico:** Su mayor beneficio es que ocultan todos los detalles técnicos de la persistencia (como consultas SQL, NoSQL o el uso de ORMs específicos), logrando que el dominio no se contamine con detalles de infraestructura[cite: 1].

### 3. Arquitecturas: CLEAN vs. Hexagonal
[cite_start]Ambos estilos arquitectónicos buscan el mismo objetivo de diseño: proteger la lógica de negocio y hacer que el código "grite su arquitectura" [cite: 1][cite_start], aplicando estrictamente la Regla de Dependencia (las dependencias siempre apuntan hacia adentro)[cite: 1].

* [cite_start]**Arquitectura Hexagonal (Puertos y Adaptadores):** [cite: 1]
    * El núcleo (dominio y aplicación) define **Puertos** (interfaces) que establecen qué puede hacer la aplicación o qué necesita del exterior.
    * La infraestructura externa provee **Adaptadores** (Controladores REST, repositorios de bases de datos, APIs de terceros) que se "conectan" a estos puertos para interactuar con el núcleo.
* **Arquitectura CLEAN:**
    * Define de forma más prescriptiva el sistema mediante capas concéntricas (Entidades en el centro, rodeadas por Casos de Uso/Aplicación, Adaptadores de Interfaz y finalmente Infraestructura externa).
    * [cite_start]Se resalta que la **Capa de Aplicación** orquesta el dominio con la infraestructura, sirviendo como una capa de acoplamiento abstracto[cite: 1].

### 4. Capas de Anticorrupción (Context Mapping)
*(Componente clave cuando los Bounded Contexts deben comunicarse)*
* **Definición (Anti-Corruption Layer - ACL):** Cuando un sistema nuevo o un *Bounded Context* central necesita comunicarse con un sistema externo, legado o de terceros que posee un modelo de datos diferente.
* **Función:** Actúa como un traductor intermediario. Se encarga de mapear o transformar los datos y conceptos del dominio externo a nuestro propio modelo interno.
* **Beneficio:** Evita que el diseño deficiente o incompatible del sistema externo "corrompa" la pureza de nuestro modelo de dominio.
