## Tema: ISO 42010 y Dimensiones de la Arquitectura de Software

### 1. Estándar ISO/IEC/IEEE 42010
El estándar ISO 42010 define las convenciones y fundamentos para la descripción de la arquitectura de sistemas y software. 
* **Propósito:** Proporcionar un marco estandarizado para la creación, análisis y mantenimiento de descripciones arquitectónicas.
* **Conceptos clave:**
  * **Sistema (System):** La entidad de interés (el software a construir).
  * **Interesados (Stakeholders):** Individuos, equipos u organizaciones con intereses en el sistema (clientes, desarrolladores, usuarios, gerentes).
  * **Preocupaciones (Concerns):** Intereses de los stakeholders sobre el sistema (seguridad, rendimiento, mantenibilidad, costo).
  * **Vistas (Views):** Una representación del sistema desde la perspectiva de un conjunto de preocupaciones relacionadas.
  * **Puntos de Vista (Viewpoints):** Las convenciones o reglas utilizadas para construir y analizar una vista específica.

### 2. Dimensiones de la Arquitectura de Software
La arquitectura no solo trata del código, sino que abarca múltiples dimensiones a lo largo del ciclo de vida del software.

#### Dimensión de Operación (Operation)
* Se refiere a cómo el sistema se ejecuta en producción.
* **Cualidades asociadas:** Rendimiento (Performance), Escalabilidad, Disponibilidad (Availability), Tolerancia a fallos, y Fiabilidad (Reliability).

#### Dimensión de Evaluación y Mantenimiento (Maintenance/Evaluation)
* Se enfoca en la capacidad del sistema para ser modificado, analizado y mejorado tras su despliegue.
* **Cualidades asociadas:** * **Mantenibilidad:** Facilidad para corregir errores o añadir nuevas funcionalidades.
  * **Testeabilidad (Testability):** Facilidad para probar el software e identificar defectos.
  * **Modularidad:** Nivel de separación de responsabilidades dentro del sistema.

#### Dimensión de Transición y Evolución (Transition)
* Involucra cómo el software cambia de un entorno a otro o cómo evoluciona en el tiempo sin romper su estructura base.
* **Cualidades asociadas:**
  * **Portabilidad:** Capacidad de mover el software a diferentes plataformas o infraestructuras.
  * **Despliegue (Deployability):** Facilidad y automatización para llevar el código desde el desarrollo hasta producción (CI/CD).

### 3. Atributos de Calidad (Quality Attributes)
Los atributos de calidad son características no funcionales del sistema (RNF - Requisitos No Funcionales) que la arquitectura debe satisfacer para que el proyecto sea exitoso. 
* Un arquitecto de software debe balancear (*Trade-offs*) estos atributos, ya que a menudo mejorar uno (ej. Seguridad) puede afectar a otro (ej. Rendimiento).
